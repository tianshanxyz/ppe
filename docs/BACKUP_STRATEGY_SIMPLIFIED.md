# 数据备份与恢复策略（简化版）

> **更新说明**: 本方案基于 Supabase 内置备份功能，无需 AWS S3 或其他外部存储。

## 一、备份架构

```
┌─────────────────────────────────────────────────────────────┐
│                     Supabase 平台                           │
│  ┌─────────────────┐    ┌──────────────────────────────┐   │
│  │  生产数据库      │    │      内置备份功能             │   │
│  │                 │───▶│  • 每日自动备份               │   │
│  │  PostgreSQL     │    │  • PITR 时间点恢复            │   │
│  └─────────────────┘    │  • 7天备份保留                │   │
│                         └──────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
                    ┌─────────────────┐
                    │  可选: 手动导出  │
                    │  (pg_dump)      │
                    │  本地存储       │
                    └─────────────────┘
```

## 二、备份策略

### 2.1 Supabase 内置备份（推荐）

| 功能 | 说明 | 频率 | 保留期 |
|------|------|------|--------|
| **自动每日备份** | 完整数据库快照 | 每日 | 7天 |
| **PITR** | 时间点恢复，可精确到秒 | 持续 | 7天 |
| **手动备份** | 通过 Dashboard 触发 | 按需 | 永久 |

**优点**:
- ✅ 零配置，开箱即用
- ✅ 无需额外费用（Pro版包含）
- ✅ 恢复简单，通过 Dashboard 操作
- ✅ 数据安全，Supabase 负责存储

### 2.2 本地备份（可选）

如需长期保留或本地存档，可手动导出：

```bash
# 使用 Supabase CLI 导出
supabase db dump -f backup_$(date +%Y%m%d).sql

# 或使用 pg_dump
pg_dump \
  -h db.xxx.supabase.co \
  -U postgres \
  -d postgres \
  -f backup_$(date +%Y%m%d).sql
```

## 三、恢复流程

### 3.1 通过 Supabase Dashboard 恢复

1. 登录 [Supabase Dashboard](https://app.supabase.com)
2. 选择项目 → Database → Backups
3. 选择要恢复的备份点
4. 点击 "Restore" 确认恢复

### 3.2 PITR 时间点恢复

1. 进入 Database → Backups → Point in Time
2. 选择恢复时间点（过去7天内任意时刻）
3. 确认恢复

### 3.3 从 SQL 文件恢复

```bash
# 使用 psql 恢复
psql \
  -h db.xxx.supabase.co \
  -U postgres \
  -d postgres \
  -f backup_20240115.sql
```

## 四、监控与告警

### 4.1 备份状态检查

```bash
# 检查最近备份（通过 Supabase API）
curl -X GET \
  'https://api.supabase.io/v1/projects/{ref}/database/backups' \
  -H 'Authorization: Bearer {token}'
```

### 4.2 健康检查

已在 `.github/workflows/health-check.yml` 中配置：
- 每日检查数据库连接
- 检查备份状态
- 失败时发送通知

## 五、RTO/RPO 目标

| 指标 | 目标值 | 实现方式 |
|------|--------|----------|
| **RPO** | 1小时 | PITR 持续备份 |
| **RTO** | 30分钟 | Dashboard 一键恢复 |

## 六、灾难恢复演练

建议每季度进行一次恢复演练：

1. **测试环境恢复**: 从备份恢复到测试环境
2. **数据完整性验证**: 检查关键表数据
3. **应用连接测试**: 验证应用可正常连接

| 本地手动导出 | $0 | 仅存储成本 |
| ~~AWS S3~~ | ~~$10-50/月~~ | ~~不推荐~~ |

## 八、总结

**推荐方案**: 完全依赖 Supabase 内置备份

- 对于大多数场景，Supabase 的自动备份 + PITR 已足够
- 如需长期存档，可每月手动导出一次到本地
- 无需额外配置 AWS S3，简化运维
